The Cognitive Architecture

Agent Memory Systems

An agent's intelligence comes from its memory architecture. Short-term memory limits what it can work with right now. Long-term memory determines what it can remember across sessions and days. Getting this architecture right is the difference between a useful agent and a forgetful one.

Short-Term Memory

Working Memory

The current conversation context window. Everything in the active prompt. Resets after each session. Limited by context window size (8K–200K tokens).

In-Session State

LangGraph's State object — messages, tool results, intermediate decisions stored across nodes within a single execution run.

Long-Term Memory

Semantic Memory

Factual knowledge retrieved via Vector DB (RAG). "What is our refund policy?" → retrieve policy doc. Scales to millions of documents. Survives across all sessions.

Episodic Memory

Log of past agent actions and their outcomes, stored as searchable records. "Last time I called this API, it rate-limited me at 100 req/min" — agent learns from experience.

Procedural Memory

Learned workflows and strategies encoded in the system prompt or fine-tuning. "For DevOps tasks, always check logs before making config changes."

Agent Planning Architectures

ReAct (Reason + Act)

Think → Act → Observe → Repeat. The agent writes its reasoning before each action, creating an observable chain of thought. Best for dynamic environments where the plan changes based on tool results.

Use for: research, exploration, dynamic tasks where next step depends on current results

Plan & Execute

A Planner LLM (high intelligence, expensive) first generates a complete task decomposition. A separate Executor LLM (cheaper, faster) carries out each step. Reduces hallucination during execution.

Use for: long multi-hour workflows, project management, coding tasks with clear deliverables

Reflection & Self-Correction

Agent generates an output, then a separate "Critic" call evaluates it against criteria. If below threshold, the agent revises. Also called Self-Refine or Reflexion. Can run multiple iterations.

Use for: writing, code review, any task where quality matters more than speed

Tree of Thoughts (ToT)

Generate 3-5 different reasoning paths simultaneously. Score each path. Prune low-scoring branches. Explore the most promising. Much higher accuracy on complex logic problems — at significant compute cost.

Use for: complex math, strategic planning, problems with many possible solution paths

Evaluating Agents: New Paradigms Required

Standard RAG metrics (faithfulness, recall) don't apply to agents. Agent evaluation measures process quality, not just output quality — because a correct final answer achieved via an unreliable, expensive path is still a bad agent.

Task Completion Rate

Did the agent achieve the final goal? Measured against golden test cases with known solutions. Most important metric.

Efficiency (Steps & Tokens)

How many tool calls and tokens did it take? Compare to a human expert's reference trajectory. An agent that takes 50 steps to do a 5-step task is inefficient and expensive.

Tool Selection Accuracy

Did it call the right tools in the right order? Compare the agent's tool call sequence against the reference sequence using trajectory metrics.

Reliability (Failure Modes)

Frequency of infinite loops, hallucinated tool calls, context overflow errors. Track over time — reliability should improve, not degrade, as the system matures.

Human Preference Rating

Domain experts rate agent trajectories on quality, approach, and judgment. Critical for tasks where "correct" is subjective (writing, strategy, code style).

Interview Prep

Top Interview Questions

What is the difference between semantic and episodic memory in agents?
Semantic memory stores facts and knowledge (what the agent knows). Episodic memory stores experiences and past events (what the agent has done). For an AI DevOps agent: Semantic memory holds documentation about Kubernetes, API schemas, and system architecture. Episodic memory holds logs of past deployments, which commands failed, which servers are flaky. Episodic memory is what turns an agent from a stateless tool into a learning system — it gets smarter with experience.
When should you use Plan & Execute vs. ReAct?
Use Plan & Execute for: long multi-step workflows (10+ steps), tasks with clear sequential dependencies, when you want a human to review the plan before execution, and when you need cost efficiency (executor uses cheaper model). Use ReAct for: exploratory tasks where the next step depends on what you find, debugging sessions where the agent needs to adapt based on error messages, and short tasks (under 5 steps) where planning overhead isn't worthwhile. Hybrid approach: use Plan & Execute but allow the executor to use ReAct within each sub-task for adaptive execution.
How do you prevent an agent from getting into an infinite correction loop?
Three strategies: (1) Hard iteration limits — set max_reflection_rounds=3; after 3 failed correction attempts, escalate to human review instead of looping again. (2) Convergence detection — track the score of each revision; if the score isn't improving by at least 5% per iteration, stop (the agent is oscillating). (3) Root cause analysis — instead of blindly retrying, have the critic identify the specific root cause of failure, so the reviser addresses the underlying issue rather than random trial-and-error. This is especially important for code generation where the same bug gets "fixed" superficially across multiple iterations.
