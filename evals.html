AI Evaluation

Mathematically prove your AI system works. Measure faithfulness, recall, precision, and quality using RAGAS, LLM-as-a-Judge, and golden test datasets.

Why You Can't Unit Test an LLM

LLMs are non-deterministic. Two identical prompts can produce different outputs. You cannot assert output == expected_string. Instead, you must statistically measure quality dimensions across hundreds of test cases using scoring rubrics. This is evaluation, not testing.

Bad Evaluation

"I ran it manually on 5 examples and it looked good." Non-reproducible, not statistically significant, misses edge cases, can't detect regressions after prompt changes.

Good Evaluation

500+ example golden dataset, automated scoring via RAGAS or LLM-as-Judge, baseline scores tracked, regression alerts trigger on 5%+ quality drop after any change.

RAGAS: The RAG Evaluation Framework

RAGAS (RAG Assessment) measures the four critical quality dimensions of RAG systems. Each metric scores 0.0–1.0. Production target: all metrics ≥ 0.80.

Faithfulness

Is every claim in the generated answer directly supported by the retrieved context? Catches hallucinations where the LLM adds information not in the chunks.

score = supported_claims / total_claims

1.0 = all facts grounded in context. 0.0 = pure hallucination.

Context Recall

Did the retrieval system find ALL the facts needed to answer the question? Measures retrieval completeness — are important facts being missed?

score = retrieved_needed_facts / all_needed_facts

Low score = your vector search is missing relevant chunks.

Context Precision

Are the retrieved chunks actually relevant to the question? Penalizes noisy retrieval — fetching chunks that are technically related but don't help answer the question.

score = relevant_retrieved / total_retrieved

Low score = your retrieval is fetching irrelevant context.

Answer Relevance

Does the generated answer actually address what was asked? Catches verbosity and off-topic responses — the model answered something adjacent but not the real question.

score = cosine_similarity(question_embedding, answer_embedding)

Low score = model answered the wrong question.

Evaluation Methodologies

Human Evaluation

Human raters score outputs on quality dimensions.

+ Most accurate, catches nuance
- Expensive, slow, doesn't scale, inconsistent

LLM-as-a-Judge

A powerful judge LLM (GPT-4, Claude Opus) scores outputs from a weaker model using a strict rubric.

+ Scalable, fast, cheap (vs. human), reproducible
- Biased toward its own style, can be gamed

Deterministic Metrics

Rule-based scoring: BLEU, ROUGE, BERTScore, exact match. No LLM needed.

+ Fully reproducible, zero cost, fast
- Misses semantic equivalence ("yes" vs "correct")

Benchmark Suites

Standardized tasks: MMLU, SWE-bench, HumanEval, MATH.

+ Cross-model comparisons, industry standard
- General benchmarks don't measure your specific use case

Building a Golden Dataset

A golden dataset is a curated set of (question, ground_truth_answer) pairs that represent your real use cases. This is the single most important asset in your AI evaluation practice. It takes time to build but pays dividends forever.

Collect real user queries (100+)
→
Write ground truth answers
→
Tag by category & difficulty
→
Run RAGAS eval suite
→
Track scores over time

RAGAS Testset Generation: RAGAS can auto-generate synthetic (question, context, answer) triples from your document corpus. Start with 50 synthetic + 50 real user examples for a balanced 100-example golden dataset.

Interview Prep

Top Interview Questions

What is LLM-as-a-Judge and what are its limitations?
LLM-as-a-Judge uses a powerful model (GPT-4, Claude Opus) to score outputs from another model against a rubric: "Rate this answer 1-10 for accuracy, completeness, and conciseness. Explain your score." It scales human-quality evaluation to thousands of examples in minutes. Limitations: (1) Self-serving bias — Claude tends to score Claude's outputs higher. (2) Verbosity bias — longer answers often score higher regardless of quality. (3) Position bias — the first option in pairwise comparisons is preferred. Mitigations: use multiple different judge models and average scores.
How do you evaluate an agentic system vs. a RAG system?
RAG evaluation focuses on output quality (RAGAS: faithfulness, recall, precision, relevance). Agent evaluation is more complex: (1) Task completion rate — did it achieve the final goal? (2) Efficiency — how many steps/tokens did it take? (3) Tool selection accuracy — did it call the right tools in the right order? (4) Reliability — how often does it loop, crash, or hallucinate tool calls? (5) Trajectory analysis — comparing the agent's step sequence to an expert's reference sequence. Use frameworks like AgentBench or WebArena for standardized agent benchmarking.
What is the difference between Faithfulness and Factual Accuracy?
Faithfulness is contextual — does the answer stay within what the retrieved chunks say? A faithful answer can still be factually wrong if the underlying documents contain errors. Factual accuracy is ground-truth — is the answer correct against real-world facts? RAGAS measures faithfulness, not factual accuracy. To measure factual accuracy, you need a golden dataset with verified ground truth answers and compare the model's claims against them. In production, both matter: faithfulness catches hallucination, factual accuracy catches knowledge gaps in your document corpus.
How do you set up a regression testing workflow for LLM changes?
Treat it like a CI/CD pipeline: (1) Maintain a frozen golden dataset in version control. (2) On every PR that touches prompts, models, or retrieval logic, automatically run the eval suite. (3) Set quality gates — fail the PR if any RAGAS metric drops more than 5% from baseline. (4) Track trend lines in a dashboard (LangSmith, Langfuse). (5) Keep a "hall of fame" baseline from your last known good state. This prevents "evaluation erosion" where each small degradation passes individually but compounds into a 30% quality drop over months.
