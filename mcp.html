Model Context Protocol

The open standard that gives AI models secure, standardized access to tools, data, and enterprise systems. Called "USB-C for AI" — build once, works everywhere.

MCP Client MCP Server Anthropic Standard STDIO / SSE Resources Tools
The USB-C of AI

The Problem MCP Solves

Before MCP, every AI tool (Cursor, GitHub Copilot, Claude Desktop) had to build custom proprietary integrations for every service (Slack, Jira, Postgres, GitHub). If 10 AI tools want to connect to 10 services, that's 100 custom integrations to maintain. MCP reduces this to 10 + 10 = 20 standard implementations.

Before MCP

Each AI tool hardcodes custom API clients per service. Cursor has its own Jira plugin. Copilot has its own Jira plugin. Neither works for the other. 10 tools × 10 services = 100 integrations.

After MCP

Build one MCP Server for Jira. Every MCP-compatible client (Claude, Cursor, Windsurf, custom agents) instantly gets Jira integration for free. 10 + 10 = 20 total implementations.

MCP Architecture: Client ↔ Server

LLM App

Claude Desktop, Cursor, your custom agent

⟷

MCP Client

Built into the host app. Manages server connections

⟷

MCP Server

Exposes your tools, resources, and prompts

⟷

Data Source

Postgres, GitHub, Slack, Filesystem, APIs

Security by Design: The LLM provider (Anthropic, OpenAI) never touches your data. The MCP Client on your machine intermediates all requests. The LLM just outputs "call this tool with these args" — your local client executes it.

What an MCP Server Exposes

Resources

Read-Only Data

File-like data the AI can read. Log files, database schemas, API documentation, configuration files. Think of these as attachable context — the AI can request to read them.

Tools

Executable Functions

Functions the AI can call to take actions. execute_sql_query, create_github_issue, send_slack_message. You control exactly what's allowed — a read-only SQL server can block INSERT/UPDATE.

Prompts

Reusable Templates

Pre-written prompt templates with arguments. "Review this PR against our security checklist." Users select these from a menu — powerful for standardizing team AI workflows.

Transport Protocols

STDIO

Standard Input/Output — the client spawns the MCP Server as a child process on the same machine. JSON-RPC messages flow over stdin/stdout. Used for local tools (filesystem, databases on localhost).

Best for: local development, desktop apps, Claude Desktop

HTTP + SSE

Server-Sent Events over HTTP. The MCP Server runs as a remote service. Enables cloud-hosted servers shared across multiple clients. Streaming responses via SSE for long-running operations.

Best for: enterprise servers, shared team infrastructure

Interview Prep

Top Interview Questions

Why was MCP created if Function Calling already existed?
Function Calling lets one model call one custom function — it requires custom code per use case per model per application. MCP is a standardized protocol for the entire client-server communication lifecycle: discovery (what tools exist?), invocation (call this tool), and result handling. An MCP Server built for Claude Desktop works identically with Cursor, Windsurf, or any MCP client — no code changes. It's the difference between building a website vs. building the HTTP standard.
How does the LLM know what tools an MCP Server has?
On startup, the MCP Client sends a `tools/list` request to the Server. The Server returns a JSON array of tool schemas: name, description, and input parameter types. The client injects these schemas into the LLM's system prompt. The LLM reads the schemas and knows what's available. This discovery is dynamic — add a new tool to the server, and all connected clients automatically see it without any code changes.
Can an MCP Server execute dangerous operations like deleting files?
Only if you program it that way. The MCP Server is code you write (or install) — you control exactly which tools it exposes. A well-designed filesystem MCP Server might only expose `read_file` and `list_directory`, blocking write operations entirely. A database server might only allow SELECT queries. This is the "least privilege" principle — give the AI only the capabilities it genuinely needs for the task at hand.
What is the difference between MCP Resources and MCP Tools?
Resources are passive, read-only data exposed as URIs (like `file:///logs/app.log` or `db://schema/users`). The AI application decides when to fetch them and attaches them to the context. Tools are active, callable functions that take arguments and produce side effects (creating, updating, deleting). Resources are like files on a network drive; Tools are like RPCs. The distinction matters for permission scoping: you might grant resource read access broadly but restrict tool write access to specific users.
How does MCP handle security when the server is remote?
Remote MCP Servers (HTTP+SSE transport) should implement standard web security: OAuth 2.0 for authentication, TLS for encryption in transit, rate limiting, and audit logging. The MCP spec includes an authorization framework. Enterprise deployments add API gateway policies (via LiteLLM or Kong) in front of MCP Servers. The key security invariant remains: the LLM provider never directly touches your server — all calls go through the client on the user's side of the trust boundary.
