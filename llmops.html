LLMOps & Observability

The operational discipline of deploying, monitoring, and maintaining LLM applications in production — managing cost, latency, quality, and drift at scale.

Telemetry Token Tracking TTFT Prompt Versioning LangSmith Caching
Production Grade

LLMOps vs Traditional MLOps

Classical MLOps focuses on model training pipelines, data drift, and model retraining cycles. LLMOps focuses on an entirely different set of problems: prompt version control, token spend management, API reliability across multiple providers, and RAG retrieval quality degradation — none of which exist in traditional ML.

Traditional MLOps

  • Model training pipeline automation
  • Feature store management
  • Model accuracy drift detection
  • Retraining triggers and schedules
  • A/B testing model versions
  • GPU training cluster management

LLMOps

  • Prompt version control & rollback
  • Token cost attribution per feature
  • API latency monitoring (TTFT, throughput)
  • RAG retrieval quality tracking
  • Provider failover & fallback routing
  • Semantic cache hit rate optimization

Core LLMOps Practices

Prompt Version Control

Treat system prompts like code — commit to git, tag releases, use PRs for prompt changes. A single word change in a system prompt can degrade accuracy by 20%. Track which prompt version is running in production and enable instant rollback.

Token Budget Management

Set hard limits: max_tokens per request, monthly API budgets per team, cost attribution tags. Implement semantic caching with tools like GPTCache to return cached responses for similar queries — dramatically reduces spend.

Latency Tracing

Instrument every step: time in queue, embedding latency, vector DB lookup, LLM TTFT, and total response time. When users complain "it's slow," traces tell you whether it's the DB, the LLM, or network — in seconds.

Provider Routing & Fallback

Use an AI Gateway (LiteLLM, Portkey) to route requests: primary model for quality, fallback to a cheaper model on timeout, local Ollama for non-sensitive data. Multi-provider setup achieves 99.99% availability vs single-provider 99.5%.

Interview Prep

Top Interview Questions

What is TTFT and why is it more important than total latency?
Time To First Token (TTFT) is the delay between sending a request and receiving the first character of the response. Humans psychologically perceive "thinking delay" as more painful than "reading time." If TTFT is under 500ms, users feel the system is responsive even if the full response takes 10 seconds — because they can read while it streams. Always stream LLM responses in production and optimize for TTFT, not total completion time.
How do you control LLM API costs at scale?
Four levers: (1) Semantic caching — detect similar queries and return cached responses without calling the API. GPTCache and Redis with embedding similarity can achieve 40-60% cache hit rates. (2) Model routing — use a cheap/fast model (Haiku, GPT-4o-mini) for simple tasks, expensive models only when needed. (3) Prompt compression — tools like LLMLingua can compress prompts 2-5x with minimal quality loss. (4) Output length caps — set max_tokens aggressively; verbose outputs waste budget.
What does "prompt drift" mean in LLMOps?
Prompt drift is when the quality of an LLM's responses degrades over time without any code change. Causes: (1) Model updates — the LLM provider silently updates the underlying model, changing its behavior. (2) Data distribution shift — the types of user queries change over time. (3) Context window pollution — prompt templates grow as features are added, crowding out important instructions. Solution: maintain a frozen golden eval dataset and run it weekly; any accuracy drop below threshold triggers an alert.
What is semantic caching, and how is it different from traditional caching?
Traditional caching does exact string matching — "What is the capital of France?" and "What's France's capital?" are different keys and get two separate API calls. Semantic caching embeds the query and looks for cached responses with high cosine similarity (threshold ~0.92). If found, return cached response instantly. This handles paraphrasing, typos, and minor variations. Tools like GPTCache and LiteLLM's built-in cache support this. For a high-traffic chatbot with lots of FAQ-style queries, 40-60% cache hit rates are common.
