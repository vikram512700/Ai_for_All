Agentic AI Systems

Build autonomous agents that plan, use tools, reflect, and solve complex multi-step problems without human intervention between each step.

ReAct Loop Tool Calling Function Calling LangGraph Autonomous
Beyond Chatbots

What Makes an AI "Agentic"?

A chatbot responds. An agent acts. An agent is an LLM given three things: a goal (what to achieve), tools (APIs, code execution, web search it can call), and a loop (it keeps running until the goal is met). The LLM becomes the brain; tools are the hands.

Chatbot

One-shot Q&A. No memory. No tools. User drives every step.

Copilot

Suggests actions. Human approves and executes each step.

Autonomous Agent

Sets sub-goals, calls tools, self-corrects, completes multi-hour tasks independently.

The ReAct Loop: How Agents Think

ReAct = Reason + Act. The agent alternates between writing its thought process and taking actions, creating an observable, debuggable trace.

1. Thought

"The user wants Apple's latest earnings. I should search the web for the Q4 2025 earnings report, then extract the revenue and EPS figures."

2. Action (Tool Call)

Outputs structured JSON calling web_search("Apple Q4 2025 earnings revenue EPS")

3. Observation

Tool returns: "Apple Q4 2025: Revenue $94.9B, EPS $1.64, beating estimates by 3%." Result injected back into context.

4. Reflection → Final Answer or Next Action

"I have enough data. Revenue was $94.9B, EPS $1.64. I can now write the report." Agent exits the loop and presents the final output.

Function Calling: The Technical Mechanism

The LLM doesn't actually call any functions — it outputs a JSON object describing which function to call and with what arguments. Your application code receives this JSON, executes the real function, and feeds the result back to the model.

"tool_use": { "type": "tool_use", "name": "web_search", "input": { "query": "Apple Q4 2025 earnings revenue EPS", "max_results": 5 } }

Your backend receives this, calls the real web_search() function, then sends the results back as a tool_result message. The model continues from there.

What Tools Can Agents Use?

web_search(query)

Search the live web via Tavily, Brave, or Serper APIs. Gives the agent up-to-date information beyond its training cutoff.

execute_code(script)

Run Python in a sandboxed interpreter. Lets the agent do math, data processing, and file manipulation programmatically.

query_database(sql)

Execute SQL against enterprise databases. Always scoped to read-only unless explicitly granted write permissions.

read_file(path)

Read files from the local filesystem or object storage (S3). Combined with write_file, enables full document editing workflows.

call_api(url, method)

Make HTTP requests to REST APIs — create GitHub issues, post to Slack, update CRM records. The real power of agents.

rag_search(query)

Query a vector database for semantically relevant documents. Combines agentic reasoning with RAG's private data access.

Core Agent Engineering Concepts

Tool Schemas

Every tool is described by a JSON schema: name, description, input parameters with types and required fields. The LLM reads these schemas and decides which tool to call. Better descriptions = better tool selection.

The ReAct Loop

Reason → Act → Observe, repeated until done. Most agents run for 5-30 iterations for typical tasks. LangGraph adds max_steps limits to prevent runaway loops from burning API budget.

Agent Memory

Working memory: The current conversation context. External memory: Vector DB or SQL for long-term recall. Agents with only working memory forget everything between sessions.

Human-in-the-Loop

Pause the agent at dangerous actions (delete file, send email, deploy code) and request explicit human approval. LangGraph's interrupt_before makes this trivial to implement.

Agent Failure Modes (and Fixes)

Infinite Loop
→
Agent retries same failing tool forever. Fix: enforce max_steps=20, add retry backoff, detect repeated tool calls in state
Tool Hallucination
→
Agent calls a tool with invalid args or fabricated tool name. Fix: strict JSON schema validation before execution
Context Overflow
→
Long agent runs fill the context window. Fix: compress observation history, summarize old steps, trim tool outputs
Prompt Injection
→
Malicious data in a tool response hijacks the agent. Fix: output guardrails, sandbox untrusted content, HITL before dangerous actions
Interview Prep

Top Interview Questions

What is the difference between Function Calling and Tool Calling?
Often used interchangeably, but technically: Function Calling is the LLM's native ability to output a JSON object matching a function signature (e.g., OpenAI's "tools" parameter). Tool Calling is the broader agentic pattern where the application actually executes that function, captures the real-world result, and feeds it back to the LLM in a loop. Function calling is the mechanism; tool calling is the loop that makes it agentic.
How does an agent know when to stop?
The agent stops when it outputs a "final answer" instead of a tool call. In LangGraph, this is implemented as a conditional edge: after each LLM call, check if the output contains a tool_use block. If yes, route to the tool execution node and loop back. If no, route to the END node. You also always add a max_steps guard to prevent runaway loops if the agent gets confused.
What is ReAct, and what problem does it solve?
ReAct (Reason + Act) is a prompting framework from a 2022 Google paper. The core insight: LLMs produce better decisions when they write their reasoning BEFORE choosing an action. Without ReAct, the model jumps straight to "call this tool" which leads to wrong tool choices. With ReAct, the model writes "I need X, I don't have it, I should use search tool" — making the decision process visible, debuggable, and more accurate.
Why use LangGraph instead of just writing a Python while-loop?
A plain while-loop works for demos. LangGraph adds: (1) Typed state management — all nodes share a structured state object, no spaghetti variable passing. (2) Checkpointing — state saved to DB after every node, enabling pause/resume and time travel. (3) Human-in-the-loop — interrupt_before hooks with clean resume semantics. (4) Streaming — observe partial outputs as the graph runs. (5) Visual debugging via LangSmith. In production, these features save weeks of custom plumbing.
What security risks do autonomous agents introduce?
Agents have tool access that chatbots don't — this amplifies every existing LLM risk: (1) Prompt injection — malicious content in a web search result can hijack the agent's tool calls. (2) Privilege escalation — an agent with write-DB access can be tricked into deleting data. (3) Exfiltration — indirect injection via documents can cause the agent to send private data to attacker URLs. Mitigations: least-privilege tool permissions, HITL gates before destructive actions, output guardrails on tool results, sandboxed code execution.
