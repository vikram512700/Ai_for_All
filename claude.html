The Model Family

Claude's Three-Tier Architecture

Anthropic ships Claude in three tiers optimized for different trade-offs. Knowing which tier to pick for a task is a key production engineering skill — using Opus for a simple classification task wastes 10x the budget.

Claude Haiku

The fastest, cheapest tier. Best for high-volume tasks: classification, summarization, simple Q&A. Ideal for real-time chat where latency < 500ms matters.

Use when: Speed > accuracy, cost per call critical

Claude Sonnet

The balanced workhorse. Consistently tops SWE-bench for coding. Best for complex reasoning, multi-file refactoring, RAG pipelines, and agentic tasks.

Use when: Production default — quality + affordability

Claude Opus

Maximum intelligence. Used for research, deep analysis, creative generation, and complex multi-step agent orchestration where accuracy is non-negotiable.

Use when: Highest accuracy, cost is secondary

Core Skill

Prompt Engineering Techniques

XML Tags

Separate data from instructions. Claude is explicitly fine-tuned on XML structure. Use <context>, <instructions>, <examples> to isolate intent from data.

Chain of Thought (CoT)

Force reasoning before output: "Think step-by-step inside <thinking> tags before answering." Every thinking token increases accuracy on logic and code tasks.

Few-Shot Prompting

Don't describe the format — show it. Provide 2-3 (input, output) pairs. The model extrapolates the exact pattern for new inputs without further explanation.

Role Assignment

Assign expertise at the top of the system prompt: "You are a senior Python engineer specializing in async FastAPI." Role priming activates domain-relevant training weights.

Output Format Control

Be explicit: "Return a JSON object with keys: summary, confidence, sources." For code: "Return only the function body with no markdown fences." Prevents post-processing.

Self-Consistency

Run the same prompt 3-5 times with Temperature >0.5 and take a majority vote. Dramatically improves accuracy for math and reasoning tasks over single-pass inference.

Production-Grade Prompt Structure

This is the structure that senior AI engineers use in real applications — not the one-liner you'd type in a chat window.

# System Prompt (set once, persists across turns) <system> You are a senior DevOps engineer specializing in Kubernetes and cloud infrastructure. You write production-ready YAML and Terraform — no placeholders, no TODOs. Always explain security implications of changes you make. </system> # User turn (data separated from instructions) <context> Existing deployment: 3 replicas, 512Mi memory limit, no HPA configured. Traffic spikes 10x every Monday at 9am UTC causing OOM crashes. </context> <instructions> Think through the problem step by step inside <thinking> tags. Then provide: 1) Updated deployment YAML with HPA, 2) Explanation of limits chosen. </instructions>

SKILL.md / CLAUDE.md: When using Claude Code CLI, create a CLAUDE.md file in your project root. Claude reads it automatically to learn your project's architecture, conventions, and style guidelines — eliminating the need to re-explain context every session.

Extended Thinking Mode

Claude's Extended Thinking feature allocates a separate token budget for internal reasoning before the final response. This is different from CoT prompting — the model genuinely reasons in a hidden scratchpad, then produces a polished final answer.

When to Enable

Complex math, multi-step logic, architecture design decisions, and tasks where you need the model to "figure it out" rather than pattern-match a memorized answer.

Cost Trade-off

Thinking tokens are billed at the same rate as output tokens. A 10,000 thinking-token budget can add $0.03–$0.15 per call. Profile before using in high-volume pipelines.

Transparency

The thinking block is returned in the API response (but not shown by default in Claude.ai). Inspect it during debugging to understand exactly where the model went wrong.

Common Prompting Mistakes (and Fixes)

Vague Role
→
"You are a helpful assistant" → "You are a senior backend engineer specializing in Go microservices"
No Format
→
"Summarize this" → "Summarize in 3 bullet points. Start each bullet with a bold keyword."
Mixed Context
→
Data embedded in instructions → Use XML tags to clearly delineate <data> from <task>
One-shot for Hard Tasks
→
Complex logic in one prompt → Break into: 1) Understand, 2) Plan, 3) Execute separate calls
Interview Prep

Top Interview Questions

Why does Claude prefer XML tags over markdown or plain text?
Anthropic explicitly trained Claude on XML-structured data. XML provides unambiguous nesting — unlike markdown, it cannot be confused with content accidentally. When you use <context>, <instructions>, and <examples> tags, Claude can reliably separate "what I should do" from "what I should do it with", even when the content itself contains markdown or other special characters.
How is Chain-of-Thought different from Extended Thinking?
CoT is a prompting technique where you ask the model to write its reasoning in the output (visible to the user). Extended Thinking is a native Claude API feature where the model allocates a hidden token budget for internal reasoning before producing the final answer. Extended Thinking is more reliable because the reasoning budget is separate from the output budget — the model won't truncate its thinking to stay within your max_tokens limit.
What is the difference between Claude Chat and Claude Code?
Claude.ai is a passive web chat interface. Claude Code is an agentic CLI tool installed via npm (`npm install -g @anthropic-ai/claude-code`). Claude Code has read/write access to your local filesystem, can execute bash commands, run tests, install packages, and operates in an autonomous loop. It reads CLAUDE.md files in your project for persistent context.
What is a System Prompt, and who controls it in a deployed app?
The system prompt is a special message set by the application developer (not the user) that defines the model's persona, constraints, and behavior. In a deployed chatbot, the user never sees it. It persists as the first message in every conversation. Prompt injection attacks specifically target the system prompt — attackers try to override it via user inputs.
What is few-shot prompting and when should you use fine-tuning instead?
Few-shot prompting gives the model 2-5 examples in the prompt context at inference time. Fine-tuning bakes those examples into the model weights during training. Use few-shot when: examples change frequently, you need flexibility, or dataset is small. Use fine-tuning when: you need consistent behavior at scale, want lower latency (shorter prompts), or are handling a high-volume narrow task where training cost amortizes over millions of calls.
How does temperature affect Claude's output in a real production scenario?
Temperature=0: deterministic, predictable — use for data extraction, classification, SQL generation where correctness matters. Temperature=0.3-0.7: balanced — use for most chat and reasoning tasks. Temperature=1.0+: creative and diverse — use for brainstorming, content generation where variety is valuable. A common mistake is using high temperature for agentic tool-use, which makes the agent unpredictably select wrong tools.
