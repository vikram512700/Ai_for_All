Prompt Engineering for Claude

Master Claude Code, understand tokens & context windows, write SKILL.md files, and connect Claude to open-source agents like LangGraph, CrewAI, and Ollama.

1. What is Claude & Claude Code?

Claude is a family of large language models built by Anthropic. Claude is designed to be helpful, harmless, and honest. Claude Code is a powerful agentic coding tool that lives in your terminal or IDE — it can read your entire codebase, edit files, run commands, search the web, and manage Git workflows autonomously.

Claude (Chat)

Conversational AI accessible via claude.ai, the API, or mobile apps. Great for brainstorming, writing, analysis, and coding assistance.

Claude Code (Agentic IDE)

An autonomous agent that runs in your VS Code or terminal. It reads files, writes code, runs tests, and commits — all while asking for your approval at each step.

2. Understanding Tokens & Context Windows

What is a Token?

A token is the smallest unit of text that an LLM processes. It is NOT always a full word. A token can be a word, part of a word, a number, or punctuation. Roughly: 1 token ≈ 0.75 words (or ~4 characters in English).

Here is how the sentence "Claude is amazing at coding!" gets tokenized:

Claude is amazing at coding !

6 tokens for 5 words. Longer or rarer words may split into multiple tokens (e.g., "Kubernetes" → ["Kub", "ern", "etes"]).

What is the Context Window?

The context window is the total number of tokens the model can "see" at once — this includes your system prompt, the conversation history, any tool results, AND the model's response. Think of it as the model's short-term memory.

⚠ Why Context Matters for Claude Code

When Claude Code reads your codebase, every file it reads consumes tokens from the context window. If you load 50 files into context, Claude may "forget" the first files by the time it finishes reading the last ones. This is why it is critical to be specific in your prompts — tell Claude exactly which files to focus on.

3. Prompt Engineering Masterclass for Claude

3.1 The Anatomy of a Great Prompt

A high-quality prompt for Claude should contain these 5 elements:

  1. Role: Tell Claude WHO it is. "You are a Senior DevOps Engineer..."
  2. Context: Give background information. "We use AKS on Azure with Terraform..."
  3. Task: Be specific about WHAT to do. "Write a Terraform module for..."
  4. Format: Specify the desired output format. "Return the output as a YAML file..."
  5. Constraints: Set guardrails. "Do NOT use deprecated APIs. Keep it under 100 lines."

3.2 Example: Bad vs Good Prompt

❌ Bad Prompt

User: Help me with Kubernetes.

Too vague. Claude doesn't know if you need a tutorial, a YAML file, a debugging session, or an architecture design.

✅ Good Prompt

User: You are a Senior Kubernetes Engineer. I have an AKS cluster (v1.29) running on Azure with 3 node pools. I need to deploy a Redis Sentinel cluster for HA caching. Write a production-ready Helm values.yaml file for the Bitnami Redis chart with: - 3 replicas - Persistent storage (50Gi, managed-csi) - Resource limits (500m CPU, 1Gi RAM) - Pod anti-affinity rules Output ONLY the YAML. No explanation.

3.3 Advanced Prompting Techniques

Chain-of-Thought (CoT)

Add "Think step by step" to force Claude to show its reasoning before giving a final answer. Dramatically improves accuracy on math and logic problems.

Few-Shot Examples

Show Claude 2-3 examples of the input/output format you expect. It will mimic the pattern perfectly for your actual request.

System Prompts

Use the system message to define Claude's persona, rules, and available tools. This is the most powerful lever for controlling behavior.

XML Tags

Claude excels at parsing structured XML tags. Wrap context in <context>...</context> and instructions in <instructions>...</instructions> for clarity.

4. Working with Claude Code (Agentic IDE)

4.1 How Claude Code Works

flowchart LR User[You Type a Prompt] --> CC[Claude Code Agent] CC --> Read[Reads Files] CC --> Search[Searches Codebase] CC --> Edit[Edits Code] CC --> Run[Runs Commands] CC --> Git[Git Commit / PR] Read --> CC Search --> CC Edit --> |Asks Permission| User Run --> |Asks Permission| User

4.2 Best Practices for Claude Code

  1. Be specific about files: Say "Edit src/auth/login.ts" not "fix the login bug".
  2. Use slash commands: Type /help to see all available commands. Use /compact to free up context.
  3. Review every change: Claude Code always asks for permission before editing or running commands. Read the diffs carefully.
  4. Use AGENTS.md: Place an AGENTS.md file in your repo root with project-specific rules (coding style, forbidden libraries, etc.). Claude Code reads this automatically.
  5. Iterate in chunks: Don't ask Claude to "refactor the entire app." Break it into component-level tasks.

5. The SKILL.md System (Custom Skills)

Skills are reusable instruction sets that extend Claude Code's capabilities for specific tasks. They live as directories in your config and are automatically loaded when relevant.

5.1 Where to Place Skills

5.2 SKILL.md Structure

A SKILL.md file has two parts: YAML frontmatter (for discovery) and a Markdown body (instructions).

---
name: terraform-azure-expert
description: Expert at writing production-grade Terraform modules for Azure infrastructure including AKS, VNET, and Key Vault.
---

# Terraform Azure Expert Skill

## Rules
- Always use `azurerm` provider version >= 3.0
- Use remote state with Azure Storage Account backend
- Follow HashiCorp naming conventions
- Include `variables.tf`, `outputs.tf`, `main.tf`, `providers.tf`
- Add inline comments for every resource block

## Coding Style
- Use `snake_case` for all resource names
- Group related resources in the same file
- Always tag resources with: environment, team, managed_by

## References
- See `references/azure-naming.md` for naming conventions
- See `examples/aks-module/` for a working AKS module

💡 Pro Tip: Trigger Matching

Skills are automatically triggered based on the name and description fields in the frontmatter. If you ask Claude Code to "write a Terraform module for Azure," it will automatically load the terraform-azure-expert skill because the description matches your request.

6. AGENTS.md (Project Rules)

While Skills are for capabilities, AGENTS.md is for project-wide rules and constraints. Place it in .agents/AGENTS.md in your repository root.

# Project Rules for AI Assistants

## Tech Stack
- Backend: Python 3.12 + FastAPI
- Frontend: Next.js 15 + TypeScript
- Database: PostgreSQL 16 with pgvector
- Cloud: Azure (AKS, Key Vault, Blob Storage)

## Coding Standards
- Use Ruff for Python linting
- Use Pydantic v2 for all data models
- Never use `print()` for logging; use `structlog`
- All API endpoints must have OpenAPI docstrings

## Forbidden
- Do NOT use SQLAlchemy ORM; use raw SQL with asyncpg
- Do NOT install new dependencies without asking
- Do NOT modify files in the `/infrastructure` folder

## Testing
- Every new function must have a pytest unit test
- Use `pytest-asyncio` for async tests
- Target 80% code coverage minimum

7.3 Lab: Claude + LangGraph Agent

from langchain_anthropic import ChatAnthropic
from langgraph.prebuilt import create_react_agent

# 1. Initialize Claude as the LLM brain
llm = ChatAnthropic(model="claude-sonnet-4-20250514")

# 2. Define tools the agent can use
def get_weather(city: str) -> str:
    """Get the current weather for a city."""
    return f"It is 22C and sunny in {city}."

def search_docs(query: str) -> str:
    """Search internal documentation."""
    return f"Found 3 results for '{query}' in company wiki."

# 3. Create the ReAct agent
agent = create_react_agent(llm, tools=[get_weather, search_docs])

# 4. Run the agent
result = agent.invoke({
    "messages": [{"role": "user", "content": "What is the weather in London?"}]
})

print(result["messages"][-1].content)

7.4 Lab: Claude + Ollama (Hybrid Local/Cloud)

Use Claude for complex reasoning and a local Ollama model for cheap, fast tasks:

from langchain_anthropic import ChatAnthropic
from langchain_ollama import ChatOllama

# Cloud model for hard tasks
claude = ChatAnthropic(model="claude-sonnet-4-20250514")

# Local model for simple tasks (FREE, no API cost)
local_llm = ChatOllama(model="llama3.2:3b")

def smart_route(query: str):
    """Route to the right model based on complexity."""
    simple_keywords = ["format", "extract", "list", "summarize"]
    if any(kw in query.lower() for kw in simple_keywords):
        print("Routing to LOCAL Ollama (free)")
        return local_llm.invoke(query)
    else:
        print("Routing to Claude API (paid)")
        return claude.invoke(query)

# Simple task -> Ollama (free)
print(smart_route("List the days of the week"))

# Complex task -> Claude (paid)
print(smart_route("Design a microservices architecture for a fintech app"))

💡 Cost Saving with Hybrid Routing

By routing simple tasks to a free local model (Ollama) and only sending complex reasoning tasks to Claude's API, enterprises can reduce their LLM API costs by 40-70% while maintaining output quality.

8. Interview Questions

Beginner

Q: What is the difference between a Token and a Word?

A: A word is a human language unit. A token is a model's processing unit. Common words like "the" are 1 token. Rare words like "Kubernetes" are split into 3 tokens. Punctuation and spaces are also tokens. Roughly 100 tokens = 75 words.

Intermediate

Q: Why does Claude sometimes "forget" things mid-conversation?

A: Because the conversation has exceeded the effective context window. While Claude supports 200K tokens, attention quality degrades with very long contexts. The solution is to periodically summarize the conversation or use RAG to retrieve only the relevant context.

Advanced

Q: How would you design a production system that uses both Claude and local open-source models?

A: Implement a Model Router: classify incoming queries by complexity using a lightweight classifier. Route simple tasks (formatting, extraction) to a local Ollama/vLLM instance running Llama-3. Route complex reasoning, code generation, and agentic workflows to Claude via the Anthropic API. Use LangGraph to orchestrate the workflow and Langfuse/LangSmith for observability across both providers.
