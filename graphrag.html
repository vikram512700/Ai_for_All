GraphRAG

Combine knowledge graphs with vector retrieval to answer multi-hop questions that require reasoning across connected entities — where standard RAG fundamentally fails.

Knowledge Graphs Entity Extraction Multi-Hop Queries Community Detection Neo4j Microsoft GraphRAG
The Core Problem

Where Standard RAG Breaks Down

Standard RAG retrieves documents by semantic similarity to the query — it finds nearby documents, not connected documents. Multi-hop questions requiring reasoning across linked entities are fundamentally unsolvable with cosine similarity alone.

Example: Multi-Hop Question That Breaks Standard RAG

Question: "What are the regulatory compliance implications of Project Falcon's data partnership with VendorX, given VendorX's recent breach history and our current SOC 2 obligations?"

This requires: Project Falcon docs → VendorX relationship → VendorX breach history → SOC 2 requirements → reasoning across 4 document clusters. Standard RAG returns top-k similar documents, but the relationships between them are invisible to cosine similarity.

A knowledge graph stores entities (nodes) and their typed relationships (edges) with metadata. Unlike a vector DB that answers "what is similar to X?", a knowledge graph answers "what is connected to X through relationship Y?"

Standard RAG vs. GraphRAG

Standard Vector RAG

  • Retrieves by cosine similarity
  • Documents are independent — no relationship data
  • Single-hop: finds docs relevant to query
  • Cannot traverse relationships
  • Struggles with global corpus themes
  • Fast to set up — chunk, embed, query

GraphRAG

  • Retrieves via graph traversal + semantic search
  • Entities and relationships are explicit, typed
  • Multi-hop: follows paths across entities
  • Excellent at relationship and reasoning queries
  • Community summaries enable global questions
  • Expensive to build — entity extraction + graph indexing

Microsoft GraphRAG: Indexing Pipeline

1
Document Chunking

Split corpus into overlapping chunks (typically 600 tokens). Standard RAG step — required before any LLM can process documents.

2
Entity & Relationship Extraction

An LLM processes each chunk and extracts: entities (name, type, description), relationships (source → type → target), and key claims. The expensive step — 1M tokens of docs can cost $50–$200 in API calls.

3
Graph Construction & Deduplication

Merge entities referring to the same object ("Sam Altman" + "the OpenAI CEO" → one node). Build the graph in Neo4j, ArangoDB, or as a local edge list.

4
Community Detection

Run the Leiden algorithm to find clusters of highly-connected entities (e.g., all AI lab entities, all regulatory entities). Generate LLM-written summary reports for each community — these enable global question answering.

5
Dual Query: Local + Global

Local search: traverse the graph from query entities. Global search: use community summaries for thematic questions. The query engine selects the strategy based on question type.

Interview Prep

Top Interview Questions

What is GraphRAG and when should you use it over standard RAG?
GraphRAG (popularized by Microsoft Research, 2024) combines knowledge graphs with LLM retrieval to answer multi-hop questions requiring reasoning across connected entities. Use it when: (1) Queries require traversing relationships ("show all subsidiaries of Company X with regulatory violations"); (2) You need global thematic questions across a large corpus ("what are the main themes in 10,000 earnings reports?"); (3) Entity relationships are valuable in your domain (biomedical, legal, financial). Don't use it when: queries are simple semantic lookups, you need fast time-to-production, or your corpus changes frequently (graph re-indexing is expensive).
How does community detection in GraphRAG enable global questions?
Global questions like "what are the main themes in all our documents?" are impossible for standard RAG — you'd need to retrieve and summarize the entire corpus. GraphRAG's community detection runs the Leiden algorithm to find tightly-connected entity clusters. For each community, an LLM generates a natural-language summary of themes, key entities, and relationships. These community summaries are indexed separately. When a global question arrives, the engine retrieves relevant summaries (not raw documents) and synthesizes an answer — turning a corpus-wide question into a manageable lookup of pre-computed summaries.
What are the main costs and limitations of GraphRAG?
Three main costs: (1) Indexing cost — entity extraction via LLM is expensive; a 10M token corpus can cost $500–$2000 in API calls just to build the initial graph. (2) Indexing time — rate-limited by the LLM; a large corpus can take hours or days. (3) Maintenance cost — when documents change, affected graph portions must be re-extracted and re-indexed. Key limitations: entity deduplication is imperfect; relationship extraction accuracy depends on LLM quality; the graph schema must be domain-specific. Best for stable, domain-specific corpora where upfront cost is amortized over many queries.
