Multi-Agent Systems

Orchestrate teams of specialized AI agents working in parallel and sequence — the architecture behind the most capable production AI systems in 2025.

Supervisor Pattern Swarm Debate Pattern A2A Protocol Error Propagation
Beyond Single Agents

Why Multi-Agent?

Problem: Context Window Ceiling

A single agent with a 200K token context window cannot process a 1M-token codebase, write a 50-page report, and monitor production logs simultaneously. One agent = one focus. Complex tasks exceed any single agent's capacity and expertise.

Solution: Specialist Division of Labor

Separate work across specialist agents: a Research Agent with web search tools, a Coding Agent with a sandboxed executor, a Writer Agent with document templates. Each focuses on what it does best, with its own context and toolset. Run them in parallel.

Supervisor Responsibilities

Receives the high-level goal. Decomposes into subtasks. Assigns each subtask to the right specialist. Monitors completion. Aggregates results. Handles partial failures by re-delegating. Does NOT execute tasks itself — only coordinates.

Worker Agent Design

Each worker receives a focused task definition, a specific toolset, and a limited context with only task-relevant info. Workers report results (success or failure + artifacts) back to the supervisor using structured schemas — not free-form text.

Coordination Patterns

Hierarchical (Supervisor)

Central supervisor delegates subtasks to workers. Clear chain of command. Easy to debug. Best for tasks with clear decomposition structure — e.g., build a SaaS feature end-to-end.

Peer-to-Peer (Swarm)

Agents communicate directly without a central coordinator. Each can hand off to any other. Very flexible. Complex to debug. Used by OpenAI Swarm for emergent task routing.

Debate Pattern

Two LLMs argue opposing positions; a judge model evaluates arguments. The winning position is adopted. Significantly improves accuracy on complex reasoning where a single LLM has blind spots.

MapReduce

Split large documents into chunks and process in parallel (map), then combine results (reduce). Classic for summarizing long documents, analyzing large codebases, or processing thousands of tickets simultaneously.

Agent Communication Protocols

ProtocolDescriptionBest For
Shared State (LangGraph) All agents read/write a shared TypedDict State via a coordinator graph. Deterministic, easy to inspect. LangGraph multi-agent subgraphs, structured workflows
A2A Protocol (Google) Agents communicate via structured JSON on a message bus. Loosely coupled, async, cross-vendor. Cross-org agent collaboration, microservice agents
HTTP Tool Calls Each agent exposes an HTTP endpoint. Supervisor calls agents as if they were tools — agent-as-a-tool pattern. Simple supervisor+worker setups, rapid prototyping
Shared DB / Vector Store Agents read/write to a shared database without direct messaging. Async access to each other's outputs. Async agents, long-running background tasks

Multi-Agent Failure Modes

Cascade Failure
→
One agent fails, supervisor treats it as global failure. Fix: design for partial failure — continue with available data and note uncertainty in output.
Conflicting Plans
→
Two workers make contradictory changes. Fix: use an immutable task ledger — each worker locks its sub-task before starting.
Token Explosion
→
Supervisor includes all worker outputs in its context — O(n) agents × O(m) tokens. Fix: workers return structured summaries + artifact IDs; full artifacts loaded only when needed.
Prompt Injection
→
Malicious content in one agent's context hijacks another's behavior. Fix: treat all inter-agent messages as untrusted — sanitize before injecting into another agent's prompt.
Interview Prep

Top Interview Questions

Why use multi-agent instead of a single agent with more tools?
Three reasons: (1) Context limits — a single agent with 50 tools wastes tokens describing tools it won't use, and can't hold history of a long task. Specialist agents have focused, minimal contexts. (2) Parallelism — a supervisor dispatches Research + Code + Writer agents simultaneously, cutting wall-clock time from O(n) sequential to O(1) parallel execution. (3) Expertise specialization — different tasks benefit from different models, prompting strategies, and tool configurations. Forcing one model with one prompt to do everything is suboptimal for all tasks simultaneously.
What is Google's A2A protocol and how does it relate to MCP?
Agent-to-Agent (A2A) is an open protocol released by Google in 2025 to standardize communication between AI agents across different vendors. It defines: (1) Agent Cards — JSON descriptors advertising what tasks an agent can accept (like OpenAPI specs for agents); (2) Task lifecycle messages (submitted/working/input-needed/completed/failed); (3) A message format supporting text, files, structured data, and streams. Relationship to MCP: they're complementary, not competing. MCP (Model Context Protocol by Anthropic) handles how a single AI model accesses tools, data sources, and resources. A2A handles how multiple agents coordinate and communicate with each other. Think of MCP as the agent's "arms" (accessing resources) and A2A as the "voice" (talking to other agents).
How do you debug a multi-agent system when something goes wrong?
Multi-agent debugging requires tracing across agent boundaries: (1) Correlation IDs — every task gets a UUID propagated through all agent calls, so you can filter LangSmith traces for a single task execution. (2) Structured logging with agent_id, task_id, input, output, tool_calls at every node — never free-form log strings. (3) LangSmith trace waterfall — shows the full hierarchy of supervisor → worker → tool calls with timing, token counts, and errors. (4) Reproduce in isolation — extract the failing worker's exact input and replay it standalone to determine if the bug is in the worker or the supervisor's delegation logic. (5) State diffs — compare LangGraph state before and after each node to identify exactly where corruption appeared.
