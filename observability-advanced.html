Why LLM Observability Is Different

Three Pillars of LLM Observability

Traditional APM tools (Datadog, New Relic) track metrics, logs, and traces for deterministic software. LLM systems add a fourth dimension: quality. Two requests with identical latency and zero errors can produce one excellent and one completely wrong answer. You need LLM-aware tooling that understands prompts, token counts, and model outputs as first-class observability primitives.

Metrics

Quantitative signals aggregated over time. TTFT (Time To First Token), tokens/request, cost/request, cache hit rate, error rate, p50/p95/p99 latency. Stored in time-series DBs (Prometheus, InfluxDB). Queried via Grafana dashboards.

Logs

Structured records of individual events. Every prompt, every completion, every tool call — logged with metadata (user_id, session_id, model, timestamp). Stored in ElasticSearch or Loki. Queryable for specific request forensics.

Traces

Distributed trace spanning the full request lifecycle: API call → prompt construction → LLM call → tool execution → response. Each span records timing, inputs, outputs. Visualized as a waterfall. The gold standard for debugging complex agent workflows.

Quality / Evals

LLM-specific fourth pillar. Online evaluation of production outputs: faithfulness scores, toxicity checks, PII detection, hallucination flags. Separate from technical metrics — a request can succeed technically while completely failing the user's intent.

Observability Tooling: What to Use When

LangSmith

LangChain Ecosystem

Deep integration with LangChain and LangGraph. Best-in-class trace waterfall visualization. Annotation tools for human review. Dataset management for offline evals. First choice if you use LangChain.

Langfuse

Open Source

Open-source, self-hostable alternative to LangSmith. Framework-agnostic — works with any LLM via SDK or OpenTelemetry. Cost analytics, prompt versioning, user feedback collection. Best for teams needing data sovereignty.

Helicone

Proxy-Based

Works as an OpenAI-compatible proxy — add one URL change to capture all requests. Zero-code integration. Cost breakdown by user, team, or project. Prompt experiments. Best for instant observability without SDK changes.

Arize / Phoenix

ML Platform

Full ML observability platform extending to LLMs. Strong on data drift detection, embedding visualization, model performance monitoring. Best for teams already using Arize for traditional ML models.

Alerting Strategy: What to Page On

P1
LLM API Error Rate > 5% (5 min window)

Model provider outage or auth failure. Immediate page to on-call. Trigger automatic failover to secondary model provider.

P1
P99 TTFT > 5 seconds (sustained > 2 min)

Severe latency degradation. Users experiencing unacceptable response times. Immediate investigation of serving infrastructure.

P2
Hourly Cost > 3× 7-day average

Abnormal token usage spike — possible runaway agent loop or cost abuse. Investigate and optionally pause affected user accounts.

P2
Cache Hit Rate drops below 20% (1 hr avg)

Semantic cache degradation — prompts no longer matching. Review recent prompt template changes or cache key configuration.

P3
Quality Score drops > 10% week-over-week

Prompt drift or model regression. Not immediate — schedule investigation within the day. Compare prompt versions and model outputs.

Interview Prep

Top Interview Questions

What are the four pillars of LLM observability and why is the fourth unique?
The four pillars are: (1) Metrics — aggregated quantitative signals (latency, cost, error rate, cache hit rate) stored in time-series DBs; (2) Logs — structured records of individual events including full prompt/completion pairs; (3) Traces — distributed spans showing the full lifecycle of a request across all services; (4) Quality/Evals — LLM-specific: online evaluation of output quality, faithfulness, toxicity, and hallucination. The fourth is unique because traditional software can be verified correct through tests alone — but LLMs are probabilistic. A request can succeed with HTTP 200 and zero latency while generating a completely wrong, harmful, or hallucinated response. Quality monitoring is the only way to catch this class of failure in production.
How do you detect prompt drift in production?
Prompt drift occurs when model behavior changes despite the same prompt template — usually because the model provider updated the underlying model, or because input data distribution shifted. Detection strategies: (1) Golden set monitoring — maintain a fixed set of 50-100 representative inputs with expected outputs; run them weekly against production and track quality scores over time. Any significant drop signals drift. (2) Output distribution analysis — track statistical properties of outputs (average response length, vocabulary diversity, sentiment scores). Sudden changes indicate model behavior change. (3) Semantic embedding drift — embed all production outputs and track the centroid over time; cluster drift signals distribution shift. (4) A/B shadow testing — keep a shadow version on the previous model checkpoint and compare response quality for the same inputs.
What is the difference between LangSmith and OpenTelemetry for LLM tracing?
OpenTelemetry (OTel) is the CNCF standard for distributed tracing across all software — it defines spans, attributes, and exporters in a vendor-neutral way. LLM providers (Anthropic, OpenAI) are adding OTel instrumentation to emit spans for every API call. OTel traces flow to any compatible backend (Jaeger, Zipkin, Datadog, Grafana Tempo). LangSmith is an opinionated, LLM-specific observability platform built on top of higher-level concepts: it understands "runs" (LangChain executions), "chains", "agents", and knows how to visualize prompt templates and compare output versions. LangSmith is a Saas product; OTel is a protocol. In practice: use OTel for baseline infrastructure traces, and use LangSmith/Langfuse on top for LLM-specific analytics, annotation, and evaluation workflows.
