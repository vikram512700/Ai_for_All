Inference Serving Frameworks

A model file on disk is useless without an inference server. The server receives HTTP requests, batches them onto GPU memory, runs the forward pass, and streams tokens back. The choice of serving framework can mean 2–5× difference in throughput and cost.

vLLM

Industry standard for GPU inference. Implements PagedAttention — serves 2–4× more concurrent requests than naive batching. OpenAI-compatible API. Multi-GPU tensor parallelism. Best throughput for most open-weight models.

PagedAttention OpenAI API Multi-GPU

NVIDIA Triton

NVIDIA's enterprise platform. Supports multiple backends: TensorRT, PyTorch, ONNX. Dynamic batching, model ensembles, A/B serving. Best for enterprise GPU clusters and maximum hardware utilization.

Multi-backend TensorRT Enterprise

Text Generation Inference (TGI)

Hugging Face's production serving library. First-class support for all HF Hub models. Continuous batching, quantization (GPTQ, AWQ), Safetensors. Easiest path from HF Hub to production. Used in HuggingChat.

HF Native Continuous Batching Streaming

Ollama / llama.cpp

CPU + consumer GPU inference. llama.cpp enables GGUF quantized models on laptops and M-series Macs. Ollama wraps it with a Docker-style CLI. Not for production scale — essential for development and edge deployment.

CPU Inference GGUF Models Dev Friendly

GPTQ — Post-Training Quantization

One-shot quantization applying a correction factor after compressing each weight. Works on any model without retraining. Standard for INT4/INT8 serving. Best quality under 13B parameters.

AWQ — Activation-Aware Quantization

Identifies the most important weights by analyzing activation magnitudes and protects them during quantization. Better quality than GPTQ at the same bit depth, especially for 70B+ models. Now the standard for INT4 production serving.

Without KV cache, every new token requires re-computing attention over the full prompt from scratch. KV cache stores key-value attention matrices in GPU memory and reuses them — turning O(n²) re-computation into O(n) incremental updates.

PagedAttention (vLLM's key innovation): KV cache is allocated in fixed-size pages — like OS virtual memory. Multiple requests share GPU memory non-contiguously, eliminating fragmentation. Traditional serving wastes ~30% VRAM in gaps; PagedAttention uses nearly all of it, enabling 2–4× more concurrent users per GPU.

Interview Prep

Top Interview Questions

What is PagedAttention and why does it matter for GPU serving?
PagedAttention is vLLM's key innovation: instead of allocating a large contiguous VRAM block per request's KV cache, it divides the KV cache into fixed-size "pages" (blocks of typically 16 tokens). Pages for different requests are scattered non-contiguously in VRAM — just like OS virtual memory. This eliminates fragmentation: traditional serving wasted ~30% of VRAM in fragmented gaps. With PagedAttention, nearly all VRAM is used efficiently, enabling 2–4× more concurrent requests per GPU — a direct 2–4× reduction in infrastructure cost per active user.
When should you use INT4 quantization vs. FP16?
Use FP16 when quality is paramount (medical, legal, financial), you have A100/H100 GPUs available, and the model is under 13B (FP16 fits fewer GPUs). Use INT4 (AWQ preferred) when deploying 70B+ models on available hardware, cost reduction is a priority, and a 1–2% quality drop is acceptable. Practical rule: run your eval suite on both quantization levels; use the cheaper one if scores are within 2%. For chat/summarization, INT4 is usually indistinguishable to humans. For complex reasoning or code, FP16 often noticeably outperforms INT4.
How does continuous batching differ from static batching?
Static batching waits for a fixed batch size before starting — if one request finishes early, the GPU idles until all batch members complete. Continuous batching (iteration-level scheduling) processes each generation step independently: once a request finishes, a new one immediately fills its GPU slot. GPU utilization goes from ~50–60% with static batching to ~80–95% with continuous batching, improving throughput 40–60%. Both TGI and vLLM implement continuous batching by default.
