Retrieval Augmented Generation (RAG)

Learn how to connect Large Language Models to your private, proprietary enterprise data using Vector Databases, embeddings, and hybrid search.

1. What is RAG?

Retrieval-Augmented Generation (RAG) is a technique that grants Large Language Models (LLMs) access to external, up-to-date, or proprietary data that they were not originally trained on. It solves the LLM hallucination problem by grounding responses in retrieved facts.

🎯 Problem it solves

LLMs have a knowledge cutoff and hallucinate when they don't know the answer. RAG forces the LLM to read relevant documents before answering, dramatically improving accuracy.

2. Real World Analogies

Human Analogy

Taking an open-book exam. Instead of relying on memory (LLM weights), you search the textbook (Retrieval), read it, then write your answer (Generation).

Cloud Analogy

Like an Azure Logic App workflow: trigger is the user prompt → query Azure AI Search → retrieve JSON → inject into Azure OpenAI call → return result.

4. Step-by-Step Technical Workflow

  1. User asks a question: "How do I configure AKS networking?"
  2. Embedding: The question is sent to an embedding model (e.g., text-embedding-3-small) which converts text into a dense vector of 1536 floats.
  3. Vector Search: Cosine Similarity search against a Vector Database (Pinecone, Azure AI Search) to find the closest matching vectors.
  4. Context Retrieval: The top-K chunks of text associated with those vectors are returned.
  5. Prompt Augmentation: Retrieved text chunks are injected into the system prompt: "Use the following context to answer..."
  6. Generation: The LLM reads the context and generates an accurate, grounded answer.

6. Common Mistakes in Production RAG

X Poor Chunking Strategy

Splitting documents every 500 tokens cuts sentences in half, destroying semantic meaning.

OK Fix: Use recursive character splitting or semantic chunking by headers/paragraphs.

X Vector Search Only

Vector search is bad at exact keyword matches like product IDs: SKU-9942.

OK Fix: Implement Hybrid Search (Vector + BM25 Keyword) and use a Cross-Encoder for re-ranking.

7. Interview Questions

🟢 Beginner

Q: What is the difference between Fine-tuning and RAG?

A: Fine-tuning permanently alters the model's weights. RAG provides external knowledge at inference time. RAG is better for factual accuracy; fine-tuning is better for stylistic alignment.

🔴 Advanced

Q: How does a Cross-Encoder Re-ranker improve RAG?

A: Standard Bi-encoder search is fast but misses deep relevance. A Cross-Encoder scores the Query AND Retrieved Document simultaneously for highly accurate ranking, but is too expensive to run on the entire index (run on top ~50 results only).

8. Hands-On Lab: Build a RAG Pipeline

from langchain_community.document_loaders import TextLoader
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain_chroma import Chroma

# 1. Load Data
loader = TextLoader("company_policy.txt")
docs = loader.load()

# 2. Embed and Store in Vector DB
vectorstore = Chroma.from_documents(docs, embedding=OpenAIEmbeddings())
retriever = vectorstore.as_retriever()

# 3. Retrieve and Generate
context = retriever.invoke("What is the remote work policy?")
llm = ChatOpenAI(model="gpt-4")
response = llm.invoke(f"Context: {context}\n\nQuestion: What is the remote work policy?")

print(response.content)
