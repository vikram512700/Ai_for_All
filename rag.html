RAG & Vector Search

Connect LLMs to private enterprise data using embeddings, vector databases, and advanced hybrid retrieval. The most deployed AI pattern in production.

Embeddings Vector DB Cosine Similarity Hybrid Search Chunking Re-ranking
The Knowledge Problem

Why LLMs Need RAG

LLMs are frozen in time after training — they know only public internet data up to a cutoff date. They cannot access your company's Confluence docs, Slack history, or internal databases. RAG (Retrieval-Augmented Generation) solves this by fetching relevant private data at query time and injecting it into the prompt context.

Without RAG

"What's our Q3 revenue?" → Model hallucinates an answer or says "I don't have that information." Useless for enterprise AI.

With RAG

"What's our Q3 revenue?" → Retrieves the Q3 report PDF chunk → "Based on the Q3 report, revenue was $14.2M, up 23% YoY." Grounded and accurate.

The 5-Step RAG Pipeline

1

Ingest & Parse Documents

Load PDFs, Word docs, Confluence pages, or database tables. Extract clean text (handle images with OCR, tables with special parsers). Tools: LlamaParse, Docling, Unstructured.io.

2

Chunk into Segments

Split documents into ~300–1000 token chunks with 10–15% overlap so sentences don't get cut mid-thought. Bad chunking is the #1 cause of poor RAG quality.

3

Embed Each Chunk

Pass each chunk through an embedding model (e.g., OpenAI text-embedding-3-large, Cohere Embed v3). Each chunk becomes a 1536-dimensional float vector representing semantic meaning.

4

Store in Vector Database

Index vectors in a specialized database that supports Approximate Nearest Neighbor (ANN) search. Also store chunk text and metadata (source, page, date) for filtering.

5

Retrieve → Augment → Generate

At query time: embed the user question → find top-K similar chunks → inject chunks into LLM prompt → LLM generates a grounded, cited answer.

Chunking Strategies

How you split your documents is the single biggest lever on RAG quality.

Chunk 1: "The capital of France is Paris..." Chunk 2: "...Paris is also the fashion capital..." Chunk 3: "...known for the Eiffel Tower..." BAD: "...Paris is also the fashion cap | ital of..."

Fixed-Size (Naive)

Split every N characters. Fast, but blindly cuts sentences and paragraphs mid-way. Only acceptable for very homogeneous structured data.

Recursive Semantic

Split on paragraphs first, then sentences, then words — only going smaller if the chunk is still too big. Best general-purpose strategy. Used in LangChain's RecursiveCharacterTextSplitter.

Document-Aware

Parse document structure (headings, sections, tables) and chunk along natural boundaries. A Markdown heading is a natural split point. Best for structured docs.

Semantic Clustering

Embed sentences individually, then cluster semantically similar consecutive sentences into one chunk. Most accurate — no artificial cuts — but computationally expensive.

Advanced RAG: Beyond Naive Retrieval

Standard

Dense Vector Search

Query → embed → find nearest vectors by cosine similarity. Works great for semantic queries ("what is our refund policy?") but fails on exact keyword matches.

Better

Hybrid Search

Combines dense vector search (meaning) with BM25 keyword search (exact terms). Scores fused via Reciprocal Rank Fusion (RRF). Best for product names, IDs, and acronyms that embeddings miss.

Best

Hybrid + Re-Ranking

After retrieving top-20 candidates, pass them to a cross-encoder re-ranker (e.g., Cohere Rerank) which scores each chunk against the query with full context. The top-5 re-ranked results are much more accurate.

Query Expansion
→
Use LLM to rewrite bad user queries into 3 better versions, then search with all 3 and merge results
HyDE
→
Hypothetical Document Embeddings — generate a fake answer, embed it, use the embedding to find real matching chunks
Parent-Child Chunks
→
Index small chunks for precise retrieval, but return the full parent section to the LLM for complete context

Lost in the Middle: Research shows LLMs perform poorly on information placed in the middle of long contexts. Always put the most relevant retrieved chunks at the very beginning or end of your RAG prompt — never bury the critical fact in the middle of 10 chunks.

Interview Prep

Top Interview Questions

Why use a Vector Database instead of a regular SQL database?
SQL performs exact keyword matching — searching "canine" won't return results for "dog." Vector databases compare mathematical distance between embedding vectors, so "dog" and "canine" land near each other in 1536-dimensional space. SQL is great for structured filters (date, price, category), which is why modern production systems combine both: use a vector DB for semantic search and filter results with SQL-like metadata constraints.
What is cosine similarity and why use it over Euclidean distance?
Cosine similarity measures the angle between two vectors, not the distance. This makes it invariant to magnitude — a short sentence and a long document about the same topic will have the same angle even though their vector magnitudes differ. Euclidean distance is affected by vector length, making it unreliable for text of varying lengths. Cosine similarity ranges from -1 (opposite) to 1 (identical), with 0.8+ indicating strong semantic relevance.
What is Hybrid Search and when would you use pure vector over it?
Hybrid search combines dense (vector) and sparse (BM25/keyword) retrieval. Use pure vector for: conversational queries, concept-level searches, multilingual content. Use hybrid for: product catalogs with SKU numbers, legal documents with exact citation references, code search where function names matter. Most production RAG systems use hybrid by default since it adds minimal latency but significantly improves recall for exact-match queries.
What is Chunking and why is it the hardest part of RAG?
Chunking splits documents into retrieval units. It's the hardest part because: (1) Too small — individual sentences lack enough context for the embedding to represent meaning accurately. (2) Too large — retrieval becomes imprecise and you waste context window tokens on irrelevant text. (3) Structure matters — a financial table split in the middle is useless. (4) No universal optimal size — PDFs, code, chat logs, and legal documents each have different natural chunk units. The standard approach is recursive semantic splitting with ~500 tokens and 10% overlap.
How does re-ranking improve RAG accuracy?
First-stage retrieval (ANN search) optimizes for speed — it finds approximate nearest neighbors using index structures. This can miss the truly most relevant chunk if it's slightly less similar by embedding distance. Re-ranking uses a cross-encoder that reads both the query and each candidate chunk together, making a holistic relevance judgment. Cross-encoders are slower (can't be pre-indexed) but much more accurate. The standard pattern: retrieve 20 candidates, re-rank to top 5, pass top 5 to the LLM.
